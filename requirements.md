# requirements.md

## 対象端末

- iPad
- iPhone
- Androidスマホ
- Androidタブレット
- PCブラウザ

## MVP要件

### キャラクター生成

- バーコード文字列からキャラクターを生成する
- 同じバーコードなら同じステータスになる
- 生成するステータス:
  - HP
  - ちから
  - ぼうぎょ
  - すばやさ

### 戦闘

- ターン制
- プレイヤーと敵が交互に行動する
- HPが0以下になったら敗北
- コマンドを選択できる

### コマンド

- たたかう
- ためる
- まもる
- 必殺

### ランダム要素

- ダメージに 0.85〜1.15 倍の乱数補正を入れる
- 必殺は命中率70%
- ただしテスト可能なように乱数注入できる設計にする

## CPU戦改善要件

### 入力バリデーション

- 空文字では戦闘を開始できない
- 短すぎるバーコード文字列では警告を表示する
- 前後の空白は無視する

### 戦闘ログ

- プレイヤーと敵の行動が分かるログを表示する
- ダメージ、ためる、まもる、必殺の命中・失敗が分かる
- 勝敗結果が分かる

### 再戦フロー

- 勝敗後に同じバーコードで再戦できる
- 勝敗後にバーコード入力へ戻れる

### 戦闘バランス

- 通常攻撃、ためる、まもる、必殺の役割が分かれる
- 極端に長すぎる戦闘や短すぎる戦闘になりにくい
- ダメージ計算はテスト可能な単位に分ける

### レスポンシブ表示

- スマホ幅でも主要操作が画面内に収まる
- コマンドボタンが押しやすい
- ステータスとログが読みやすい

## 2人ローカル対戦要件

### モード選択

- CPU戦と2人ローカル対戦を選べる
- 既存のCPU戦は引き続き遊べる

### キャラクター生成

- プレイヤー1とプレイヤー2がそれぞれバーコードを入力する
- 各プレイヤーのバーコードからキャラクターを生成する

### ローカル対戦

- 1台のブラウザで2人対戦できる
- 各プレイヤーが交互にコマンドを選択する
- 相手が選択する前に自分のコマンド内容が見えない
- 既存の戦闘ロジックを再利用する

## 将来要件

- QRコード参加
- カメラでバーコード読み取り
- ランキング
- アカウント機能

## Firebase同期対戦要件

### 対戦部屋

- ホストが対戦部屋を作成できる
- 参加者が部屋IDで参加できる
- Phase 3ではQRコード参加は対象外とし、部屋IDの手入力で接続する
- 1部屋はホスト1人、ゲスト1人の2人対戦に限定する

### 同期方式

- GitHub Pagesで配信できる静的Webアプリとして実装する
- 別ブラウザ間の同期にはFirebase Realtime Databaseを使う
- 自前のExpressサーバー、Socket.IOサーバー、WebRTCシグナリングサーバーはPhase 3では作らない
- 部屋状態、参加者状態、キャラクター準備状態、コマンド選択状態、ラウンド結果をFirebase上で同期する
- 各プレイヤーは自分のブラウザで自分のコマンドだけを選ぶ
- 相手が選んだコマンドは、両者の選択が揃うまで表示しない
- 両者のコマンドが揃ったら既存の戦闘ロジックでラウンドを解決する

### 同期と公平性

- ホスト側をラウンド解決の権威側とする
- ゲストは自分のコマンド選択状態を書き込み、ホストがFirebase上の状態を監視してラウンドを解決する
- ホストはラウンド結果をFirebaseへ書き込み、両者が同じ結果を読む
- Phase 3では後出し防止のcommit/revealは対象外とし、次フェーズ以降の検討事項にする
- 乱数はホスト側で発生させ、解決結果を両者へ同期する

### 通信状態表示

- 接続待ち、接続済み、切断の状態を表示する
- 自分の選択済み、相手の選択待ちが分かる表示にする
- 通信エラー時は対戦画面を壊さず、入力または部屋作成へ戻れるようにする
- Phase 3では切断時の自動再接続、観戦、複数部屋一覧は対象外とする
