# requirements.md

## 対象端末

- iPad
- iPhone
- Androidスマホ
- Androidタブレット
- PCブラウザ

## MVP要件

### キャラクター生成

- バーコード文字列からキャラクターを生成する
- 同じバーコードなら同じステータスになる
- 生成するステータス:
  - HP
  - ちから
  - ぼうぎょ
  - すばやさ

### 戦闘

- ターン制
- プレイヤーと敵が交互に行動する
- HPが0以下になったら敗北
- コマンドを選択できる

### コマンド

- たたかう
- ためる
- まもる
- 必殺

### ランダム要素

- ダメージに 0.85〜1.15 倍の乱数補正を入れる
- 必殺は命中率70%
- ただしテスト可能なように乱数注入できる設計にする

## CPU戦改善要件

### 入力バリデーション

- 空文字では戦闘を開始できない
- 短すぎるバーコード文字列では警告を表示する
- 前後の空白は無視する

### 戦闘ログ

- プレイヤーと敵の行動が分かるログを表示する
- ダメージ、ためる、まもる、必殺の命中・失敗が分かる
- 勝敗結果が分かる

### 再戦フロー

- 勝敗後に同じバーコードで再戦できる
- 勝敗後にバーコード入力へ戻れる

### 戦闘バランス

- 通常攻撃、ためる、まもる、必殺の役割が分かれる
- 極端に長すぎる戦闘や短すぎる戦闘になりにくい
- ダメージ計算はテスト可能な単位に分ける

### レスポンシブ表示

- スマホ幅でも主要操作が画面内に収まる
- コマンドボタンが押しやすい
- ステータスとログが読みやすい

## 2人ローカル対戦要件

### モード選択

- CPU戦と2人ローカル対戦を選べる
- 既存のCPU戦は引き続き遊べる

### キャラクター生成

- プレイヤー1とプレイヤー2がそれぞれバーコードを入力する
- 各プレイヤーのバーコードからキャラクターを生成する

### ローカル対戦

- 1台のブラウザで2人対戦できる
- 各プレイヤーが交互にコマンドを選択する
- 相手が選択する前に自分のコマンド内容が見えない
- 既存の戦闘ロジックを再利用する

## Phase 4: カメラ読み取り要件

- スマホのカメラでバーコードを読み取って入力欄に反映できる
- 読み取り結果は既存のバーコード入力検証を通す
- カメラ権限が拒否された場合や未対応ブラウザでは、手入力に戻れる
- 通信対戦のバーコード入力欄から使える
- カメラ読み取りはバーコード入力の補助機能として扱う
- 小さいバーコードや黒以外のバーコードの読み取り率を上げるため、複数の画像補正候補を試す
- 端末が高解像度カメラや大きいCanvasに対応しない場合でも、低い設定へフォールバックして読み取りを継続する
- Android実機調査中はデバッグログを通常表示する
- 調査不要な場合は、明示的にデバッグログを無効化できる
- カメラ読み取り結果は、キャラクター生成に使える文字列として数字のみ8桁または13桁か検証する
- 仕様に合わない読み取り結果は入力欄へ反映せず、エラー表示せずにスキャンを継続する
- 採用対象は数字8桁または13桁のみとする
- ISBNはEAN-13として読み取れる `978` / `979` 始まりの13桁として、文字列条件に合う場合だけ採用対象に含める
- CODE 39、CODE 93、Codabar、CODE 128、ITF、GS1 DataBar系はPhase 4では採用せず、読み取れても入力欄へ反映しない
- 手入力も数字のみ8桁または13桁に限定する

## Firebase同期対戦要件

### 対戦部屋

- ホストが対戦部屋を作成できる
- 参加者が部屋IDで参加できる
- Phase 3ではQRコード参加は対象外とし、部屋IDの手入力で接続する
- 1部屋はホスト1人、ゲスト1人の2人対戦に限定する

### 同期方式

- GitHub Pagesで配信できる静的Webアプリとして実装する
- 別ブラウザ間の同期にはFirebase Realtime Databaseを使う
- 自前のExpressサーバー、Socket.IOサーバー、WebRTCシグナリングサーバーはPhase 3では作らない
- 部屋状態、参加者状態、キャラクター準備状態、コマンド選択状態、ラウンド結果をFirebase上で同期する
- 各プレイヤーは自分のブラウザで自分のコマンドだけを選ぶ
- 相手が選んだコマンドは、両者の選択が揃うまで表示しない
- 両者のコマンドが揃ったら既存の戦闘ロジックでラウンドを解決する

### 同期と公平性

- ホスト側をラウンド解決の権威側とする
- ゲストは自分のコマンド選択状態を書き込み、ホストがFirebase上の状態を監視してラウンドを解決する
- ホストはラウンド結果をFirebaseへ書き込み、両者が同じ結果を読む
- Phase 3では後出し防止のcommit/revealは対象外とし、次フェーズ以降の検討事項にする
- 乱数はホスト側で発生させ、解決結果を両者へ同期する

### 通信状態表示

- 接続待ち、接続済み、切断の状態を表示する
- 自分の選択済み、相手の選択待ちが分かる表示にする
- 通信エラー時は対戦画面を壊さず、入力または部屋作成へ戻れるようにする
- Phase 3では切断時の自動再接続、観戦、複数部屋一覧は対象外とする

## Phase 5: 通信対戦向けプロフィールとランキング

### 既存モードの扱い

- CPU戦と2人ローカル対戦は開発工程上の足場として扱い、通常画面の導線から外す
- Phase 5時点では既存ドメインロジックをすぐに削除せず、通信対戦が安定した後に不要コード整理を検討する
- 通常ユーザー向けの主導線は通信対戦に絞る

### 簡易プロフィール

- 本格的な認証は実装しない
- ユーザー名を登録、保存、変更できる
- ユーザー名は同じブラウザ内で維持する
- 未設定の場合は通信対戦開始前に入力を促す
- ユーザー名は空文字を不可とし、長すぎる名前や表示を壊す文字列を受け付けない
- 通信対戦の部屋、準備、対戦、結果画面で自分と相手のユーザー名を表示する

### ランキング

- ランキング対象は通信対戦の勝敗結果に限定する
- CPU戦と2人ローカル対戦の結果はランキングへ登録しない
- 勝利数、敗北数、対戦数、最終対戦日時を記録する
- ランキングはFirebase Realtime Databaseへ保存し、別ブラウザのユーザーと共有できるようにする
- ランキング保存に失敗しても対戦結果画面を壊さず、再試行または通常操作へ戻れる
- Phase 5では不正対策や本人確認は簡易扱いにし、強い認証やサーバー検証は非対象にする

## Phase 6: 画面フロー改善

### 目的

- Phase 5で1画面に並んだプロフィール、部屋作成、部屋参加、ランキングを、ユーザーの進行に沿った画面フローへ整理する
- 通信対戦を主導線とし、初回ユーザーでも次に何をすればよいか分かる状態にする

### 画面フロー

1. タイトル
2. ユーザー名入力
3. ロビー
   - バーコード読み込み
   - 部屋を作る
   - 部屋に参加する
4. 待機・準備
   - 部屋ID表示
   - 相手待ち
   - 自分と相手の準備状態
5. 対戦
6. 勝利、敗北、引き分けの結果
7. ランキング

### 画面要件

- ユーザー名未設定時は、タイトルとユーザー名入力を主表示にする
- ユーザー名保存後にロビーへ進める
- ロビーでは自分のバーコードを読み込み、キャラクターを準備できる
- キャラクター準備後に部屋作成または部屋参加へ進める
- 部屋作成後は部屋ID、相手待ち、準備状態を表示する
- 対戦終了後は結果画面を明確に表示し、ランキングへ進める
- ランキング画面からロビーへ戻れる
- エラー発生時は現在の画面を壊さず、復帰操作を提示する

### 非対象

- 本格認証
- QRコード参加
- EAN/UPC以外のバーコード形式への拡張対応
- CPU戦と2人ローカル対戦の未使用コード削除
- ランキングの不正対策

## Phase 7以降

- QRコード参加
- EAN/UPC以外のバーコード形式への拡張対応
- CPU戦と2人ローカル対戦の未使用コード削除
- Firebase匿名認証または外部認証
- ランキングの不正対策
- WebRTC DataChannel対戦は対象外にする
